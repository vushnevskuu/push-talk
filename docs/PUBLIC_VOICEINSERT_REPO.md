# Два репозитория: приватный код и публичные анонсы

## Зачем

- **Приватный** (`push-talk` или как у тебя называется) — исходники, CI, секреты в настройках Actions, внутренние ветки.
- **Публичный** — только то, что видят все: README, changelog, анонсы релизов, ссылки на сайт и скачивание. Релизы GitHub в публичном репо могут отдавать ZIP всем; в приватном репо релизы видны только тем, у кого есть доступ.

## Что сделать на GitHub (вручную)

1. Создай **новый репозиторий** → **Public**, имя напр. `push-talk-public` или `VoiceInsert`.
2. Не клонируй туда этот же код: в публичном репо **нет** исходников приложения.
3. Локально:

   ```bash
   mkdir push-talk-public && cd push-talk-public
   git init
   cp "/path/to/голосовое управление/docs/templates/public-repo-README.md" README.md
   # отредактируй ссылки под свой домен и org
   git add README.md && git commit -m "Initial public landing for VoiceInsert"
   git branch -M main
   git remote add origin https://github.com/<ORG>/<PUBLIC_REPO>.git
   git push -u origin main
   ```

4. По желанию добавь `CHANGELOG.md`, папку `announcements/` (один файл на релиз или на месяц).

## Скачивание приложения

- **Сайт** (рекомендуется для покупателей): положи `VoiceInsert-macos.zip` в `web/public/releases/` (или CDN) и дай кнопку на лендинге.
- **Публичный репо**: вкладка Releases → прикрепляй тот же ZIP вручную или отдельным workflow (см. ниже).

Приватный репозиторий по-прежнему может собирать релиз тегом `v*` для **команды**; публичную ссылку для всех держи на сайте или в публичном репо.

## Автоматизация (позже)

Чтобы CI **приватного** репо пушил релиз в **публичный**, нужен Personal Access Token (classic) с `repo` на публичный репозиторий, сохранённый как секрет приватного репо, и отдельный job — это не обязательно на старте.

## Про «постить всё туда»

Ассистент в Cursor может **подготовить** текст анонса / changelog; **закоммитить и запушить** в публичный репо должен человек (или настроенный CI), потому что у ассистента нет постоянного доступа к твоему GitHub.
